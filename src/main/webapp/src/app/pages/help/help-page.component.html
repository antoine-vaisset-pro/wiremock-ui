User Documentation

WireMock UI is a visual interface for managing a WireMock OSS server. Use the table of contents on the left (or the search bar above) to find what you need.

Dashboard

The Dashboard gives you a real-time overview of your WireMock instance's activity. Data refreshes automatically every 30 seconds; you can also force a refresh with the button.

Statistics cards

Active Stubs
Total number of stub mappings currently registered in WireMock.
Total Requests
Number of HTTP requests received during the selected time range.
Unmatched
Requests that did not match any stub in the selected time range. Click the card to navigate to the Requests page pre-filtered to unmatched requests.
Recording
Current recording status: Never Started, Recording or Stopped. Click to navigate to the Recording Studio.

Time Range selector

Choose the period over which requests and statistics are computed. Available presets: 5m, 15m, 30m, 1h, 3h, 6h, 12h, 24h, 2d, 7d, 30d, and a Custom date/time range. The granularity of the chart adapts automatically to the selected range.

Requests Over Time chart

An interactive line/bar chart (Highcharts) showing the volume of requests per time unit. Hover over a data point to see the exact count. The granularity adapts to the selected time range (e.g., per minute for short ranges, per hour or day for longer ones).

Top 5 Endpoints

The five most frequently called URL paths during the selected period, with their call counts. Clicking an endpoint navigates to the Requests page filtered on that path.

Unused Stubs

Stubs that have never been matched in the selected time range. Use this list to identify dead stubs that may be cleaned up. Clicking a stub opens its details in the Stubs page.

Stubs

The Stubs page is the central place to manage all your WireMock stub mappings. It is split into a list panel on the left and a details panel on the right.

Browsing & searching stubs

Filter stubs…
Real-time search across URL, HTTP method, stub name and JSON body. The list updates as you type.
Pagination
20 stubs per page. Use the < / > arrows or click a page number to navigate.
Select all / Deselect all
Checkbox buttons in the toolbar to batch-select stubs for export or deletion.

Stub detail panel

Click any stub in the list to open its details on the right. The panel shows:

  • Details tab — formatted view of the request matcher and response configuration.
  • JSON tab — raw JSON as WireMock stores it.
  • Edit — opens the creation/edition modal pre-filled with this stub's data.
  • Clone — creates a duplicate of the stub (UUID is regenerated).
  • Delete — deletes the stub after confirmation.

Creating & editing a stub

Click + Create Stub in the top-right corner, or Edit on an existing stub. The modal offers two modes:

Simple mode (guided form)

Stub Info
Name
Optional human-readable label shown in the list.
Priority
Integer (default 5). Lower number = higher priority. When multiple stubs match the same request, the one with the lowest priority wins. See WireMock docs → Stub Priority.
Persistent
If checked, the stub survives a "Reset all stubs" operation and is written to disk.
Request Matching
Method
HTTP verb: GET, POST, PUT, DELETE, PATCH, HEAD, OPTIONS, ANY.
URL — Exact
Full path + query string must match exactly. Maps to url in the WireMock JSON.
URL — Path
Only the path is matched exactly (query string is ignored). Maps to urlPath.
URL — Pattern
Path matched as a regular expression. Maps to urlPathPattern.
URL — Regex
Full URL (path + query) matched as a regular expression. Maps to urlPattern.
Query Parameters optional

Add one or more query parameter matchers. Each matcher has:

Name
The query parameter key (e.g. page).
Match type
equalTo, contains, matches (regex), doesNotMatch, absent. See WireMock docs → Query Parameter Matching.
Value
Expected value (or regex pattern for matches/doesNotMatch). Not required for absent.
Case insensitive
When checked, comparison ignores letter case.
Request Headers optional

Match specific HTTP request headers using the same match types as query parameters.

Body Patterns optional

Match the request body using one or more patterns. Multiple patterns are combined with AND logic. See WireMock docs → Body Matching.

equalToJson
Structural JSON comparison. Options: Ignore array order and Ignore extra elements.
matchesJsonPath
A JSONPath expression that must match a value in the body.
equalToXml
Structural XML comparison.
matchesXPath
An XPath expression that must match.
contains
The body must contain the given substring.
matches / doesNotMatch
The body must (or must not) match the given regular expression.
Cookies optional

Match request cookies by name using the same match types.

Basic Auth optional

Match HTTP Basic Authentication credentials (username + password).

Form Parameters optional

Match application/x-www-form-urlencoded body parameters by key.

Response — Direct mode
Status
HTTP response status code (e.g. 200, 404).
Body Type
JSON, XML, Text, Binary (Base64) or None.
Body
The response body content. For JSON/XML, basic syntax validation is applied.
Response Headers
Add arbitrary response headers (key/value). Content-Type is auto-set based on Body Type.
Delay
Fixed delay in milliseconds, or a Random delay (uniform distribution between min/max ms), or a Chunked Dribble delay to simulate slow transfer.
Response Templating
Enables Handlebars templating in the response body and headers. Use {{ '{' }}{{ '{' }}request.url{{ '}' }}{{ '}' }}, {{ '{' }}{{ '{' }}request.headers.Authorization{{ '}' }}{{ '}' }}, {{ '{' }}{{ '{' }}jsonPath request.body '$.id'{{ '}' }}{{ '}' }}, etc.
Response — Proxy mode

Forward matching requests to a real upstream API. See WireMock docs → Proxying.

Proxy Base URL
The upstream server to forward requests to (e.g. https://api.example.com).
Additional Proxy Headers
Headers added to the proxied request.
Remove Proxy Headers
Headers removed from the proxied request before forwarding.
Response — Fault mode

Simulate network-level faults. Available faults: CONNECTION_RESET_BY_PEER, EMPTY_RESPONSE, MALFORMED_RESPONSE_CHUNK, RANDOM_DATA_THEN_CLOSE. See WireMock docs → Simulating Faults.

Scenario State optional

Bind this stub to a WireMock scenario for stateful behaviour.

Scenario Name
Identifies the scenario this stub belongs to. All stubs in the same scenario share state.
Required State
The state the scenario must be in for this stub to be matched (e.g. Started).
New State
The state to transition the scenario to after this stub is matched.

Advanced mode (raw JSON)

Toggle to Advanced to edit the WireMock stub JSON directly. The JSON must conform to the WireMock admin API schema. Switching back to Simple mode will attempt to parse the JSON back into the form.

Export & Import

Export selected
Downloads a .json file containing only the checked stubs.
Export all
Downloads all stubs as a single JSON file (WireMock mappings format).
Import stubs
Upload a .json or .zip file. Stubs in the file are merged with existing ones (existing UUIDs are updated, new ones are added).
Delete selected
Permanently removes all checked stubs after confirmation.

Deep linking

The URL reflects the current selection and search query, e.g. /ui/stubs/<uuid>?q=search. You can share or bookmark these URLs.

Scenarios

WireMock Scenarios let you model stateful behaviour: different stubs respond based on the current "state" of a named scenario. A common use case is simulating a REST resource that can be created, updated and deleted in sequence.

Scenario list

Each row shows a scenario name, its current state, and the list of stubs attached to it. The initial state is always Started.

Available actions

Reset
Resets a single scenario back to Started.
Reset all
Resets every scenario to its initial state at once.
View stubs
Navigates to the Stubs page pre-filtered on the stubs belonging to the scenario.
Define scenario stubs in the Stub creation modal → Scenario State section. All stubs sharing the same Scenario Name participate in the same state machine.

Requests

The Requests page shows all HTTP requests received by WireMock, with filtering, match details, and navigation to related stubs.

Filters

Status
Matched — request was handled by a stub.
Unmatched — no stub matched (WireMock returned 404).
Near-miss ≥ 90 % — no exact match but at least one stub scored ≥ 90 % similarity.
Method
Multi-select: GET, POST, PUT, DELETE, PATCH, HEAD, OPTIONS.
Response code
Filter by HTTP status class: 2xx, 3xx, 4xx, 5xx.
Time range
Last Hour, Last 24h, Today, Last 7 days, or a Custom date/time interval.

Request detail

Click any request to expand its details:

  • Request — method, URL, headers (expandable), body.
  • Response — status code, response headers, body.
  • Matched stub — link to the stub that handled this request (if matched).
  • Near-misses — for unmatched requests, the closest stubs with their similarity score.

Actions

Create stub from request
Opens the Stub creation modal pre-filled with the request's method, URL and headers.
Select multiple
Check requests and delete them in bulk.
Delete all
Clears the entire WireMock request log.

Deep linking

Active filters and the selected request are encoded in the URL (e.g. /ui/requests/<requestId>?status=unmatched), making the view bookmarkable and shareable.

Requester

The built-in HTTP client lets you send requests directly to your WireMock instance (or any other server) to verify that your stubs behave as expected — without leaving the UI.

Request form

Pre-fill from stub
Select any existing stub to auto-populate Method, URL, headers and body from the stub's request definition.
Base URL
The server root (e.g. http://localhost:8080). Defaults to the WireMock URL configured in Settings.
Method
HTTP verb: GET, POST, PUT, DELETE, PATCH, HEAD, OPTIONS.
URL path
Path appended to the Base URL (e.g. /api/users).
Headers
Add/remove arbitrary request headers. Click + Add Header to add a row; click to remove one.
Body
Free-text request body (JSON, XML, plain text, etc.). Shown only for methods that support a body (POST, PUT, PATCH).

Response panel

Status
HTTP status code and reason phrase returned by the server.
Timing
Round-trip time in milliseconds measured by the browser.
Response headers
All headers returned by the server.
Body
Response body with syntax highlighting. Use the Copy button to copy it to the clipboard.
You can also reach the Requester pre-filled from a request log entry: click Send to Requester in the Requests page detail panel.

Recording Studio

The Recording Studio captures real HTTP interactions against a target API and automatically converts them into WireMock stubs. See WireMock docs → Record & Playback.

Status banner

Displays the current recording state (Never Started, Recording… or Stopped) and provides contextual action buttons while recording is active:

Snapshot
Saves the stubs recorded so far without stopping the recording session.
Stop
Stops the recording and saves all captured stubs into WireMock.

Basic configuration

Target Base URL
The real upstream API to record from (e.g. https://api.example.com). WireMock will proxy requests to this URL while recording.
Persist stubs to disk
When enabled, generated stubs are written to the WireMock mappings/ folder (they survive server restarts).
Manage repeats as scenarios
When the same endpoint is called multiple times with different responses, WireMock automatically creates a scenario to replay them in order.

Advanced configuration

URL Filter Pattern
A regex applied to request URLs. Only matching requests are recorded (e.g. /api/.*).
Method Filter
Only record requests for the selected HTTP method (leave blank for all).
Capture Headers
Comma-separated list of request headers to include as matchers in the generated stubs (e.g. Authorization, Content-Type).
Text Size Threshold
Maximum size (bytes) of a text response body to capture inline. Larger bodies are stored as body files.
Binary Size Threshold
Same as above for binary (non-text) response bodies.

Saved configurations

Recording configurations can be saved locally (in the browser's localStorage) for reuse. Use the Save As… button to name and persist the current configuration, Load to restore it, and Delete to remove it.

Recorded stubs

After stopping (or taking a snapshot), the captured stubs are displayed as cards. Click View on a card to inspect the full JSON. Use Export All to JSON to download all recorded stubs.

OpenAPI Generator

The OpenAPI Generator converts an OpenAPI / Swagger specification into ready-to-use WireMock stub mappings in three steps: import the spec, configure each endpoint, then preview and export.

1 Import OpenAPI Specification

Provide your OpenAPI spec (JSON or YAML, versions 2.0, 3.0 and 3.1 are supported) in one of two ways:

Paste content
Paste the raw JSON or YAML directly into the text area.
Upload file
Drag & drop or browse for a .json, .yaml or .yml file. The parsed content is displayed in a read-only preview below the drop zone.

Click Parse Specification to analyse the spec. A summary banner shows the spec version, API title, version number and the number of operations found. Any parsing warnings are displayed below the banner.

2 Configure Endpoints

Each operation found in the spec appears as a row in the endpoint table. Use the checkboxes on the left to include or exclude individual endpoints (or use Select all at the top).

Global options

URL Prefix
A prefix prepended to every path when generating the stub URL matcher (e.g. /api/v1). Type the value and click Apply — it is propagated to all endpoints immediately.
Generate a WireMock fault variant
When checked, each endpoint will also generate a fault stub (using the fault type chosen in the Fault column). The fault type can still be overridden per endpoint.

Per-endpoint columns

Method
HTTP method badge (GET, POST, PUT, DELETE, PATCH…).
Path
The operation path as defined in the spec. Paths with {param} placeholders are automatically converted to WireMock urlPattern regex matchers.
Summary
The summary field from the spec, used as the stub name.
Status
HTTP response status code for the generated stub. Defaults to the first 2xx response defined in the spec. Automatically updated when you choose a response example that belongs to a specific status code.
Req. Example
Optionally select a named request body example defined in the spec (requestBody.content.*.examples). When selected, the generated stub will include a bodyPatterns: [{ equalToJson: … }] matcher so that only requests with that exact body will match. Shows — if the operation has no named examples.
Resp. Example
Optionally select a named response example defined in the spec (responses.*.content.*.examples). The dropdown shows key (statusCode) for each example across all response codes. Choosing an example automatically updates the Status column to match and uses that example's value as the response body. Shows — if no named examples exist.
URL Prefix
Per-endpoint URL prefix override (same as the global prefix but editable individually).
Fault
Combobox for generating an additional WireMock fault stub for this endpoint. Select — none — (default) to skip, or pick one of the four fault types: CONNECTION_RESET_BY_PEER, EMPTY_RESPONSE, MALFORMED_RESPONSE_CHUNK, RANDOM_DATA_THEN_CLOSE. When a fault is selected, the Status, Req. Example and Resp. Example columns are greyed out (they are irrelevant for fault stubs).
Path parameters (/users/{id}) are matched with a regex pattern (urlPattern: ^/users/[^/]+(\?.*)?$). These stubs are automatically assigned a lower priority (5) so that more specific stubs take precedence.

3 Preview & Export

Click Generate Stubs to produce the WireMock JSON mappings. The result list shows each stub with its method, URL and status code.

Actions on the stub list

Preview
Opens a modal with the full WireMock JSON mapping for that stub. The JSON can be copied to the clipboard from the modal header.
Download
Downloads the individual stub as a .json file.

Export & Import actions

Import into WireMock
Sends each generated stub directly to the configured WireMock Admin API (POST /__admin/mappings), one by one. A success banner shows the number of stubs imported. Partial failures are reported with the first error message.
Download ZIP Archive
Packages all generated stubs into a ZIP file with the WireMock directory structure: mappings/ for the JSON stub files and __files/ for large response bodies that exceed the inline limit (2 KB).

Body generation rules

The response body is filled using the following priority order:

  1. The selected named example (examples[key].value) if one is chosen in the Resp. Example column.
  2. The first entry of the examples object in the media type.
  3. The example field of the media type entry.
  4. The example or default field of the JSON schema.
  5. A minimal generated object derived from the schema definition (types, properties, enums, formats).

Stubs with a response body smaller than 2 KB use inline jsonBody; larger bodies are stored in __files/ and referenced via bodyFileName.

Use Reset (top-right of the page) to clear the current spec and start over. All generated stubs, configurations and previews are discarded.

Settings

The Settings page configures which WireMock Admin API endpoint the UI communicates with. The active endpoint is also selectable at any time from the server selector in the top-right corner of the app — changing it reloads the current page with data from the new backend.

WireMock API Endpoint

The endpoint list is displayed in a compact format: Name : URL badge. Options are always shown in the following order: Local → ENV → Custom. Select any row with the radio button to activate it; the choice is persisted in localStorage.

Local
The built-in local endpoint that proxies to WireMock running on the same host (path /__admin resolved relative to the UI server).
ENV Environment endpoints
URLs injected at build/deploy time via the WIREMOCK_BACKEND environment variable (semicolon-separated list of base URLs, e.g. https://staging.example.com;http://localhost:8080). These appear with an ENV badge and cannot be edited from the UI.
custom Custom backends
Backends you have saved in the Custom Backends section below. Each entry appears in the list with a custom badge and is instantly selectable.
Enter URLs without the /__admin suffix — e.g. http://my-server:9090 instead of http://my-server:9090/__admin. The UI appends /__admin automatically to all Admin API calls.

Custom Backends

Use this section to save, edit or delete named WireMock backend URLs for later reuse. Each entry requires a label (displayed in the endpoint list and the header selector) and a base URL (without /__admin).

Add
Fill in the label and URL fields, then click Add. The new backend immediately appears in the endpoint list and the header selector.
Edit
Inline editing of label and URL. If the edited backend was active, the active URL is updated automatically.
Delete
Removes the backend. If it was the active endpoint, the UI falls back to the first available option.
