## [Version 1.3.1.8] - 2026-02-12

### ✨ Amélioration Majeure : Chargement Préalable des Near-Misses

**Problème résolu** : Les filtres "Unmatched" et "Near-miss >90%" avaient des problèmes de chargement dynamique et d'interférence.

**Solution** : Chargement des near-misses dès le chargement de la page au lieu de le faire à la demande.

**Modifications** :

**1. Chargement automatique au démarrage** :
```typescript
loadRequests(): void {
  // Charge les requests
  this.requestService.getRequests().subscribe({
    next: (response) => {
      this.requests = response.requests;
      // Charge immédiatement les near-misses pour toutes les unmatched
      this.loadNearMissesForAllUnmatched();
    }
  });
}

loadNearMissesForAllUnmatched(): void {
  const unmatchedRequests = this.requests.filter(req => !req.wasMatched);
  unmatchedRequests.forEach(request => {
    this.requestService.getNearMissesForRequest(request).subscribe({
      next: (response) => {
        this.nearMissCache.set(request.id, response.nearMisses);
      }
    });
  });
}
```

**2. Simplification du filtre** :
```typescript
// Plus besoin de chargement dynamique, juste vérifier le cache
if (this.statusFilter === 'near-miss-90') {
  filtered = filtered.filter(req => {
    if (req.wasMatched) return false;
    const nearMisses = this.nearMissCache.get(req.id) || [];
    return nearMisses.some(nm => (1 - nm.matchResult.distance) >= 0.9);
  });
}
```

**3. Suppression du code complexe** :
- ❌ Supprimé `loadingNearMissCache` (plus nécessaire)
- ❌ Supprimé `nearMissCacheLoaded` (plus nécessaire)
- ❌ Supprimé `loadNearMissCacheForUnmatched()` (remplacée)
- ❌ Supprimé spinner de chargement dynamique
- ❌ Supprimé CSS de chargement

**Avantages** :

- ✅ **Simplicité** : Logique de filtrage beaucoup plus simple
- ✅ **Fiabilité** : Pas d'interférence entre filtres
- ✅ **Performance** : Chargement en parallèle au démarrage
- ✅ **UX** : Pas d'attente au clic sur le filtre
- ✅ **Maintenabilité** : Code plus court et clair

**Résultat** :
- ✅ Filtre "Unmatched" fonctionne parfaitement
- ✅ Filtre "Near-miss >90%" fonctionne parfaitement
- ✅ Pas de saut visuel
- ✅ Pas de spinner compliqué
- ✅ Cache toujours à jour

**Fichiers modifiés** :
- `requests-page.component.ts` : Logique complètement simplifiée
- `requests-page.component.html` : Bouton simplifié (plus de spinner)
- `requests-page.component.css` : CSS de chargement supprimé

## [Version 1.3.1.7] - 2026-02-12

### 🐛 Bug Fix Critique : Filtres "Unmatched" et "Near-Miss" Corrigés

**Problèmes identifiés** :
1. Le filtre "Near-miss >90%" ne fonctionnait toujours pas correctement
2. Le filtre "Unmatched" ne fonctionnait plus (régression)
3. Le texte "(loading...)" créait un saut visuel

**Causes** :
- Logique de filtrage "near-miss" retournait `!this.nearMissCacheLoaded` qui interférait avec les autres filtres
- CSS avec transitions `max-width` créait des problèmes de layout
- Ordre des opérations incorrect dans le filtre

**Solutions appliquées** :

**1. Correction de la logique de filtrage** :
```typescript
// Déclenche le chargement AVANT le filtrage
if (!this.nearMissCacheLoaded && !this.loadingNearMissCache) {
  this.loadNearMissCacheForUnmatched();
}

// Pendant le chargement : affiche toutes les unmatched
if (this.loadingNearMissCache) return true;

// Après chargement : filtre selon le cache
if (this.nearMissCache.has(req.id)) {
  return nearMisses.some(nm => (1 - nm.matchResult.distance) >= 0.9);
}

// Si pas dans le cache après chargement : ne pas afficher
return false;
```

**2. Simplification du CSS** :
```css
.filter-chip .loading-text {
  display: none;  /* Caché par défaut */
}

.filter-chip .loading-text.visible {
  display: inline;  /* Affiché quand visible */
  margin-left: 3px;
}
```

**Résultat** :
- ✅ Filtre "Unmatched" fonctionne correctement (affiche toutes les requêtes non matchées)
- ✅ Filtre "Near-miss >90%" fonctionne correctement (affiche uniquement celles >90%)
- ✅ Pas de saut visuel au clic
- ✅ Pas d'interférence entre les filtres
- ✅ Logique claire et prévisible

**Fichiers modifiés** :
- `requests-page.component.ts` : Logique de filtrage corrigée
- `requests-page.component.css` : CSS simplifié (display: none/inline)

## [Version 1.3.1.6] - 2026-02-12

### 🐛 Bug Fix : Saut Visuel du Bouton Near-Miss

**Problème** : Le bouton de filtre "Near-miss >90%" avait un comportement visuel étrange au premier clic - il devenait très gros puis se réduisait brutalement.

**Cause** : Le texte "(loading...)" apparaissait avec `*ngIf`, ce qui ajoutait du contenu au DOM et faisait grossir le bouton instantanément.

**Solution** :
- ✅ Remplacement de `*ngIf` par `[class.visible]` pour le texte "(loading...)"
- ✅ Le texte est toujours dans le DOM mais masqué avec `opacity: 0` et `max-width: 0`
- ✅ Transition smooth avec `opacity` et `max-width` quand il devient visible
- ✅ Ajout de classe `.filter-chip-text` pour le texte principal

**Modifications CSS** :
```css
.filter-chip .loading-text {
  opacity: 0;
  max-width: 0;
  overflow: hidden;
  transition: opacity 0.2s ease, max-width 0.2s ease;
}

.filter-chip .loading-text.visible {
  opacity: 1;
  max-width: 100px;
}
```

**Résultat** :
- ✅ Pas de saut visuel au clic
- ✅ Apparition smooth du texte "(loading...)"
- ✅ Taille du bouton stable
- ✅ Meilleure expérience utilisateur

**Fichiers modifiés** :
- `requests-page.component.html` : `*ngIf` → `[class.visible]`
- `requests-page.component.css` : Ajout transitions opacity + max-width

### 🐛 Correction Critique : Toutes les Requêtes Jusqu'à `endTime` Sont Maintenant Comptées

**Problème** : Certaines requêtes n'étaient pas comptabilisées dans le graphique, même si elles étaient dans la période filtrée. Par exemple, avec 20 requêtes récupérées, seulement 18 étaient affichées.

**Cause** : Lorsque `endTime` tombait au milieu d'un intervalle (par exemple 14:30 dans un intervalle d'1 heure), `roundToInterval()` arrondissait **vers le bas** (14:00), et l'intervalle qui contenait réellement `endTime` (14:00-15:00) n'était pas créé.

**Exemple concret** :
```
endTime : 2026-02-12 14:30
Intervalle : 1 heure
roundToInterval(14:30) = 14:00

lastInterval créé : 14:00
Requêtes entre 14:00 et 14:30 : arrondies à 14:00
→ Intervalle 14:00 existe ✓

MAIS si une requête arrive à 14:35 :
→ Arrondie à 14:00 
→ Mais 14:35 > endTime (14:30) donc filtrée ✗

Le vrai problème : les requêtes de la dernière portion 
de l'intervalle ne sont pas comptées.
```

**Solution** : Ajouter un intervalle supplémentaire si `endTime` n'est pas exactement sur une borne d'intervalle.

```typescript
let lastIntervalTimestamp = roundToInterval(endTime);

// Si endTime n'est pas exactement sur un intervalle, ajouter un intervalle de plus
if (lastIntervalTimestamp < endTime) {
  lastIntervalTimestamp += intervalMs;
}
```
